Privacy Filter (Anonymization) Documentation

Redaction & DPDP Compliance - Complete Technical & User Documentation

1. Summary

The Privacy Filter (Anonymization) service is a comprehensive medical document anonymization engine that detects and removes Personally Identifiable Information (PII) and Protected Health Information (PHI) from a wide range of file formats including DICOM medical images, NIfTI volumes, standard images (PNG, JPG, TIFF, BMP), PDFs, and text documents.

The service employs a multi-layered detection strategy combining metadata analysis, connected-component text region detection, OCR (Tesseract/PaddleOCR), regex-based PHI classification, and Safe Harbor pattern matching. It supports three redaction methods: masking (black rectangles), cropping (border removal), and inpainting (appearance-preserving fill). Post-redaction validation ensures no residual PHI remains.

3. Architecture

3.1 Processing Pipeline - Images/DICOM/NIfTI

  1. File Loading: LoaderRegistry routes to DicomLoader, NiftiLoader, or ImageLoader based on file extension. Returns a MedicalArtifact object with image data, metadata, and format information.
  2. Metadata Detection: MetadataDetector scans DICOM tags, EXIF data, or NIfTI headers for known PHI fields (PatientName, PatientID, PatientBirthDate, InstitutionName, etc.).
  3. Metadata Cleaning: DicomCleaner strips 15+ PHI tags and regenerates UIDs. NiftiCleaner clears descrip, aux_file, db_name fields. EXIFCleaner removes EXIF, IPTC, XMP, and PNG text chunks.
  4. Overlay Detection: Modality-specific edge band heuristics detect high-contrast overlay regions.
  5. Text Region Detection: Connected-component analysis on corner crops (top 5-15% of image height). Detects character blobs (4-60px height, 2-120px width) and groups into text lines. All detected text in corners is treated as PHI regardless of content.
  6. OCR Detection: Tesseract or PaddleOCR reads text with word-level bounding boxes.
  7. PHI Classification: Regex and keyword rules classify OCR text into PHI categories (NAME, DATE, IDENTIFIER, etc.). Confidence threshold: 0.50.
  8. Redaction: MaskRedactor (black rectangles with 3px padding), CropRedactor (remove PHI-heavy borders, max 35% per side), or InpaintRedactor (OpenCV appearance-preserving fill).
  9. Post-Redaction Validation: Re-scan redacted image via OCR + PHI classification. Risk score = min(residual_count x 5, 100). Passes if no residual PHI detected.
  10. Output: Save via SaverRegistry (DicomSaver, NiftiSaver, ImageSaver) preserving original format.

3.2 Processing Pipeline - PDFs

  1. PDF Loading: PyMuPDF (fitz.open) loads PDF, renders pages to 150 DPI images for OCR fallback.
  2. Native Text Extraction: Extract text with word-level coordinates from PDF structure.
  3. Safe Harbor Detection: 18 HIPAA Safe Harbor identifier categories - names (with honorifics), DOB, dates, times, ages, contacts (email/phone), identifiers (PIN, barcode, long numbers, alphanumeric codes).
  4. Character-to-Word Mapping: Map detected character spans to word-level rectangles for precise redaction.
  5. QR Code Detection: Aspect ratio 0.8-1.25, max side 160pt. Barcode detection: AR > 2.5, height < 70pt.
  6. Global Repeat Pass: Search known PHI tokens across all pages (names, IDs found on any page are searched globally).
  7. PDF Metadata Scrubbing: Remove author, title, creator, producer, dates, XMP data.
  8. Redaction Application: PyMuPDF redaction annotations with black rectangles.
  9. Scanned Page Fallback: Pages with < 5 native words fall back to OCR-based detection.

4. API Reference

4.1 Authentication

All redaction endpoints require a Bearer JWT token in the Authorization header. To obtain a token: Log in to the web portal and generate your token from the API Access tab within the Privacy Filter service. Direct per-service token generation endpoints (/api/demo-token) are disabled in production.

  • Algorithm: HS256 JWT signed with SECRET_KEY
  • Default expiry: 1 day
  • Usage: Include in all requests as Authorization: Bearer <token>

5. Security

  • All processing occurs locally - no document data sent to external services
  • DICOM UID regeneration prevents cross-referencing with original studies
  • Metadata stripping removes all identifying tags before output
  • Post-redaction validation ensures no residual PHI remains
  • Session logger stores only metadata - never the document content or detected PHI text
  • Compliant with HIPAA Safe Harbor de-identification standard (18 identifier categories)

6. User Guide

6.1 Browser Usage

  1. Navigate to the DPI Platform website
  2. Click the 'Privacy Filter (Anonymization)' tab
  3. Upload a file (DICOM, NIfTI, image, PDF, or text document)
  4. View detected entities in the entities table (text, type, confidence, position)
  5. Compare original vs. redacted preview side-by-side
  6. Optionally draw additional manual redaction boxes on the preview
  7. Download the redacted file

6.2 API Usage (curl)

Step 1 - Obtain a Token: Visit the web portal, log in, and generate your API token from the API Access tab.

Step 2 - Sync redaction:

curl -X POST http://localhost:8003/api/redact -H "Authorization: Bearer <YOUR_TOKEN>" -F "file=@scan.dcm"

Step 3 - Download redacted file:

curl -o redacted.dcm http://localhost:8003/api/files/redacted/<KEY> -H "Authorization: Bearer <YOUR_TOKEN>"

6.3 API Usage (Python)

import requests

TOKEN = "YOUR_TOKEN"
headers = {"Authorization": f"Bearer {TOKEN}"}

# Submit file for redaction
with open("scan.dcm", "rb") as f:
    resp = requests.post(
        "http://localhost:8003/api/redact",
        headers=headers,
        files={"file": f}
    )
data = resp.json()
print("Redacted file URL:", data.get("redacted_url"))

6.4 Local Execution (Offline Mode)

The Privacy Filter can be run locally using Docker Compose. No cloud APIs or external services are required — all OCR and redaction processing runs on your machine.

  1. Prerequisites: Docker and Docker Compose installed on your machine.
  2. Clone the repository: git clone https://github.com/TANUH-AI/TANUH-DPI.git
  3. Start the required services:
docker compose up redis privacy-filter celery-privacy-worker
  1. Access the service: Open http://localhost:8003 in your browser, or use the API endpoints directly.
  2. Disable authentication (optional for local dev): Set KEYCLOAK_AUTH_ENABLED=false in your .env file to skip token verification.

All processing occurs locally — no document data leaves your machine. Temporary files are automatically purged after processing.

8. DICOM Metadata Handling

8.1 PHI Tags Stripped

PatientName, PatientID, OtherPatientIDs, PatientBirthDate, PatientAge, PatientAddress, AccessionNumber, StudyID, InstitutionName, ReferringPhysicianName, OperatorsName, PerformingPhysicianName, StationName, DeviceSerialNumber, ProtocolName.

8.2 UIDs Regenerated

StudyInstanceUID, SeriesInstanceUID, SOPInstanceUID - regenerated via pydicom.uid.generate_uid() to prevent cross-referencing with original studies.

8.3 De-identification Flag

The DICOM dataset is marked as de-identified (PatientIdentityRemoved = YES) after processing.

9. Known Limitations & Future Enhancements

  • No burned-in pixel annotation OCR for DICOM (relies on corner text detection)
  • Inpainting redaction is slower than masking (2-3x)
  • Large NIfTI volumes (500+ slices) may approach the 10-minute task timeout
  • Planned: GPU-accelerated OCR, ML-based PHI detection model, batch processing support
