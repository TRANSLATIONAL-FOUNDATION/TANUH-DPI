Clinical Document Documentation

FHIR Parsing & Conversion - Complete Technical & User Documentation

1. Summary

The pdf2abdm service is a core microservice within the DPI Integrated Platform that converts clinical PDF documents (Diagnostic Reports and Discharge Summaries) into HL7 FHIR R4 Document Bundles compliant with ABDM (Ayushman Bharat Digital Mission) and NHCX profiles.

The service is built on a privacy-first architecture, using LLM-orchestrated dynamic workflows powered by Google Gemma 4 (26B parameters) via Vertex AI.

Key capabilities include multi-engine OCR extraction, intelligent document classification, dependency-aware FHIR resource generation using LangGraph workflows, and rulebook-driven structural enforcement.

2. Service Overview

2.1 Purpose

The service addresses the challenge of converting unstructured clinical PDF documents into structured, interoperable FHIR data. It enables healthcare systems to digitize paper-based clinical records into standardized ABDM-compliant bundles that can be exchanged across the Indian healthcare ecosystem.

2.2 Key Features

  • Privacy-first architecture: all processing runs on-premises with zero external data leakage
  • Multi-engine OCR waterfall: PyPDF > Docling > LightOn for optimal text extraction
  • Intelligent document classification: two-tier keyword + LLM gate
  • LangGraph dynamic workflow: dependency-aware, per-resource FHIR extraction
  • Rulebook-driven generation: ABDM StructureDefinition JSONs enforce correct structure
  • Multi-patient grouping: automatically groups pages in multi-patient lab batch reports
  • Original PDF embedding: base64-encodes the source PDF into DocumentReference
  • Synchronous and asynchronous processing modes
  • FHIR validation via HL7 Validator CLI
  • Prometheus metrics and structured logging

3. Architecture

3.1 System Architecture

The pdf2abdm service operates within the DPI Docker Compose stack. It exposes a FastAPI HTTP API on port 8000 and delegates long-running processing to Celery workers consuming from the dedicated 'abdm' Redis queue.

Architecture components:

  • FastAPI API Server (pdf2abdm container, port 8000) - handles HTTP requests, file validation, sync processing, and task submission
  • Celery Workers (celery-abdm-worker, 4 replicas x concurrency 2 = 8 parallel slots) - execute the full OCR-to-FHIR pipeline asynchronously
  • Redis (port 6379) - serves as Celery broker, result backend, and task status cache
  • Vertex AI MaaS - provides LLM inference for classification, resource extraction, and distillation
  • Session Logger (port 8002) - receives fire-and-forget audit logs for every processing event

3.2 Processing Pipeline

The end-to-end pipeline from PDF upload to FHIR Bundle output follows these stages:

  1. File Upload & Validation: Accept PDF (max 25 MB, max 100 pages), validate format
  2. OCR Text Extraction: Multi-engine waterfall - PyPDF (fast, native text), then Docling (LaTeX-based layout), then LightOn (fast OCR) if needed. Threshold: if PyPDF extracts < 50 chars/page average, fall back to next engine.
  3. Multi-Patient Grouping: For lab batch reports, detect multiple patients by grouping pages by (Age/Sex, Collection Date) fingerprint. Non-lab documents are treated as single-patient.
  4. Document Classification Gate: Two-tier classification - Tier 1 uses keyword heuristics (< 1ms), Tier 2 falls back to LLM if inconclusive. Returns CLINICAL, INSURANCE, or INVALID. Non-CLINICAL documents are rejected with HTTP 422.
  5. Document Type Classification: LLM analyzes text to select between DischargeSummaryRecord or DiagnosticReportRecord. Determines mandatory and optional FHIR resources to extract.
  6. LangGraph Dynamic Workflow: Resources are topologically sorted by dependency order. A LangGraph workflow is dynamically constructed with one node per resource type. Each node calls the LLM with the resource's rulebook (StructureDefinition JSON) as structural constraint.
  7. Per-Resource LLM Extraction: Each FHIR resource is extracted independently via dedicated LLM calls.
  8. Sanitization & Normalization: Fix hallucinated fields, ensure UUIDs, set NDHM ABDM profile URLs, enforce correct statuses and required fields.
  9. Bundle Assembly: Composition resource placed first (index 0), remaining resources in dependency order, DocumentReference last. All entries get fullUrl as urn:uuid:{id}.
  10. PDF Embedding: Original PDF base64-encoded and injected into DocumentReference.content[0].attachment.data.
  11. Audit Logging: Fire-and-forget POST to session-logger with processing metadata.

4.2 Authentication

All processing endpoints require a Bearer JWT token in the Authorization header.

To obtain a token: Log in to the web portal and generate your token from the API Access tab within the Clinical Document service. Direct per-service token generation endpoints (/api/token) are disabled in production.

  • Algorithm: HS256 JWT signed with ABDM_SECRET_KEY
  • Claims: sub, name, email, type, service ("pdf2abdm"), iat, exp
  • Default expiry: 1 day (configurable via ABDM_TOKEN_EXPIRY_DAYS)
  • Usage: Include in all requests as Authorization: Bearer <token>

5. Security

5.1 Authentication

The service supports two authentication mechanisms:

Demo Tokens (HS256): JWT tokens issued via the web portal's API Access tab. Signed with HMAC-SHA256 using the ABDM_SECRET_KEY environment variable. Suitable for demos, hackathons, and testing.

Keycloak Tokens (RS256): Enterprise SSO tokens issued by an external Keycloak identity server. Validated against the Keycloak realm's JWKS endpoint. Enabled when KEYCLOAK_REALM_URL is set. Suitable for production deployments.

5.2 Data Privacy

  • All document processing occurs on-premises within the Docker stack
  • No clinical data is sent to external services (LLM inference uses Vertex AI within the GCP project)
  • GCS uploads are optional and fire-and-forget (non-fatal if disabled)
  • Temporary files are deleted after processing
  • Session logger stores only metadata (filename, processing time, IP) - never the document content
  • Compliant with Digital Personal Data Protection (DPDP) Act requirements

5.3 Known Limitations

  • No token revocation mechanism (tokens remain valid until expiry)

9. FHIR Output Format

The service generates FHIR R4 Document Bundles compliant with ABDM/NDHM profiles. Each bundle contains:

  • Bundle.resourceType = 'Bundle', Bundle.type = 'document'
  • Composition resource at entry[0] (DischargeSummaryRecord or DiagnosticReportRecord)
  • Patient, Practitioner, Organization resources
  • Clinical resources (Condition, Procedure, Observation, DiagnosticReportLab, etc.)
  • DocumentReference with embedded base64 PDF as last entry
  • All entries linked via UUID references (fullUrl: urn:uuid:{id})
  • Meta.profile set to NDHM ABDM profile URLs

The output can be validated against the HL7 FHIR Validator with IG nrces.in.ndhm#6.0.0 using the POST /validate endpoint.

10. User Guide

10.1 Browser Usage

  1. Click the 'Clinical Document' tab
  2. On the API Access sub-tab, enter your name and email to request a token
  3. Return to the main upload tab
  4. Drag and drop or select a clinical PDF file
  5. Wait for processing (progress bar shows current stage)
  6. View the generated FHIR Bundle in the results panel
  7. Download the JSON bundle or validate it against HL7 standards

10.2 API Usage (curl)

Step 1 - Obtain a Token: Visit the web portal, log in, and generate your API token from the API Access tab.

Step 2 - Submit a PDF (async):

curl -X POST http://localhost:8000/pdf2abdm/submit -H "Authorization: Bearer <YOUR_TOKEN>" -F "file=@report.pdf"

Step 3 - Poll status:

curl http://localhost:8000/pdf2abdm/task-status/<TASK_ID> -H "Authorization: Bearer <YOUR_TOKEN>"

Step 4 - Get result:

curl http://localhost:8000/pdf2abdm/task-result/<TASK_ID> -H "Authorization: Bearer <YOUR_TOKEN>"

10.3 API Usage (Python)

import time
import requests

TOKEN = "YOUR_TOKEN"
headers = {"Authorization": f"Bearer {TOKEN}"}

# Submit PDF
with open("report.pdf", "rb") as f:
    resp = requests.post("http://localhost:8000/pdf2abdm/submit", headers=headers, files={"file": f})
task_id = resp.json()["task_id"]

# Poll until complete
while True:
    result = requests.get(f"http://localhost:8000/pdf2abdm/task-result/{task_id}", headers=headers).json()
    if result.get("status") in ["completed", "failed", "rejected"]:
        print(result)
        break
    time.sleep(2)

10.4 Local Execution (Offline Mode)

A standalone CLI executable is available for fully offline clinical document processing. No Python, cloud APIs, or network connectivity required — all data stays on your machine.

  1. Install Ollama: Download from ollama.com/download. On Linux: curl -fsSL https://ollama.com/install.sh | sh
  2. Pull the LLM model (~16 GB): ollama pull gemma4:26b — requires NVIDIA GPU with 16 GB+ VRAM or Apple Silicon with 16 GB+ unified memory.
  3. Download the executable: Go to the Run Locally tab on the platform and download the binary for your OS (Linux x86_64, Linux ARM64, Windows, or macOS).
  4. Unzip and make executable: On Linux/Mac: chmod +x nhcx-extract*
  5. Run:
# Verify setup
./nhcx-extract check -m gemma4:26b

# Process a clinical document
./nhcx-extract abdm report.pdf -o result.json -m gemma4:26b

The output JSON contains a complete FHIR R4 Bundle identical to the cloud service output.

11. Known Limitations & Future Enhancements

11.1 Current Limitations

  • High latency due to multi-step LLM extraction (30-120 seconds per document)
  • No automated FHIR validation correction loop (planned: LLM-in-the-loop)
  • Large rulebooks stress the LLM context window
  • No email verification for token issuance

11.2 Planned Enhancements

  • LLM-in-the-loop validation: iterative FHIR validator → LLM fix cycle (3-4 iterations)
  • Missing field resolver: compare extracted text vs generated JSON, regenerate missing resources
  • Terminology & coding fixer: validate SNOMED, LOINC, UCUM codes via LLM
  • Support for additional clinical artifact types
