Insurance Document Extraction Service

pdf2nhcx — NHCX InsurancePlanBundle FHIR Generator Complete Technical & User Documentation

1. Summary

The pdf2nhcx service is a core microservice within the DPI Integrated Platform that converts insurance policy PDFs into HL7 FHIR R4 Collection Bundles compliant with NHCX (National Health Claims Exchange) standards. It supports 6 bundle types: InsurancePlanBundle, ClaimBundle, ClaimResponseBundle, CoverageEligibilityRequestBundle, CoverageEligibilityResponseBundle, and TaskBundle.

The service uses a unique 'Text Distillation' pre-processing stage that condenses verbose insurance documents into actionable fact sheets before extraction, dramatically improving LLM extraction accuracy. It runs as a FastAPI application on port 8001, with asynchronous processing via Celery workers on the dedicated 'nhcx' queue.

2. Service Overview

2.1 Purpose

Insurance policy documents contain complex financial rules, multi-level benefit structures, waiting period logic, exclusion clauses, and legal formatting. The service transforms these unstructured PDFs into machine-readable FHIR resources that can interoperate with the Indian health claims ecosystem under NHCX standards.

2.2 Key Features

  • Privacy-first architecture with zero external data leakage
  • Insurance Text Distillation: 4 parallel LLM calls split and condense text into fact sheets
  • Support for 6 NHCX bundle types (InsurancePlan, Claim, ClaimResponse, CoverageEligibility, Task)
  • LangGraph dependency-aware workflow orchestration
  • NHCX-specific rulebook enforcement (StructureDefinition JSONs)
  • Synchronous and asynchronous processing modes
  • FHIR validation via HL7 Validator CLI

3. Architecture

3.1 Processing Pipeline

  1. File Upload & Validation: Accept PDF (max 25 MB, max 100 pages)
  2. OCR Text Extraction: Same multi-engine waterfall as pdf2abdm (PyPDF > Docling > LightOn)
  3. Document Classification Gate: Two-tier keyword + LLM classification. Rejects CLINICAL and INVALID documents (only accepts INSURANCE).
  4. Text Distillation: Split raw OCR text into 4 chunks with 2000-character overlap. Run 4 parallel LLM calls via ThreadPoolExecutor. Each call condenses its chunk into an 'Insurance Fact Sheet' — removing narrative fluff, preserving financial data, limits, exclusions, benefit tables.
  5. NHCX Resource Selection: LLM analyzes distilled text to determine the primary bundle type (1 of 6) and select mandatory + optional FHIR resources.
  6. LangGraph Dynamic Workflow: Build dependency-ordered graph. Each node extracts one resource using LLM with NHCX-specific rulebook and bundle-type-specific context prefix.
  7. Bundle Assembly: assemble_nhcx_collection_bundle() — flatten nested Bundles, strip ABDM/forbidden resources, sanitize hallucinated fields, deduplicate by ID, enforce NRCES profile URLs.
  8. Result Storage & Logging: Same as pdf2abdm (Redis 24h TTL, GCS upload, session-logger)

3.2 Text Distillation (Key Differentiator)

Insurance documents are typically 20-100 pages of dense legal and financial text. Feeding this directly to an LLM exceeds context limits and produces poor results. The distillation stage solves this:

  • Raw text is split into 4 roughly equal chunks with 2000-character overlap at boundaries
  • 4 parallel LLM calls (ThreadPoolExecutor, max 4 workers) condense each chunk
  • Each LLM call acts as an 'underwriter' that extracts only actionable insurance data
  • Output: concatenated fact sheet with financial limits, exclusions, benefit tables, waiting periods
  • This reduces context size by 60-80% while preserving all critical information

4. API Reference

4.1 Authentication

Identical mechanism to pdf2abdm. POST /pdf2nhcx/api/token with {name, email} to get a 1-day HS256 JWT. Signed with NHCX_SECRET_KEY. Include as Authorization: Bearer in all requests.

5. Deployment

Docker Compose: docker-compose up --build (starts all services including pdf2nhcx on port 8001)

Standalone: uvicorn pdf2nhcx.app.main:app --host 0.0.0.0 --port 8001

6. User Guide

6.1 Browser Usage

  1. Navigate to the DPI Platform website
  2. Click the 'Insurance Policy' tab
  3. Request an API token via the API Access sub-tab
  4. Upload an insurance policy PDF
  5. Wait for processing (distillation takes ~60-90s, total ~60-180s)
  6. View and download the generated NHCX FHIR Bundle

6.2 API Usage (curl)

# Get token

curl -X POST http://:8001/pdf2nhcx/api/token -H "Content-Type: application/json" -d '{"name": "Name", "email": "email@example.com"}'

# Submit PDF

curl -X POST http://:8001/pdf2nhcx/submit -H "Authorization: Bearer " -F "file=@policy.pdf"

# Poll and fetch result

curl http://:8001/pdf2nhcx/task-result/ -H "Authorization: Bearer "

The distillation step (4 parallel LLM calls) accounts for approximately 40-50% of total processing time. The LangGraph extraction accounts for another 40%. OCR and assembly are relatively fast.

8. FHIR Output Format

The service generates FHIR R4 Collection Bundles (Bundle.type = 'collection') compliant with NHCX/NRCES profiles.

  • Primary resource placed first in Bundle.entry
  • Supporting resources in dependency order
  • DocumentReference and Binary resources placed last
  • All entries linked via UUID references
  • Meta.profile set to NRCES NHCX profile URLs
  • Forbidden resources stripped (e.g., Condition not allowed in InsurancePlanBundle)

10. Known Limitations & Future Enhancements

  • High latency due to distillation + multi-step extraction
  • No automated validation correction loop
  • Very large policies may stress LLM context windows even after distillation
  • No rate limiting on API endpoints
  • Planned: LLM-in-the-loop FHIR validation cycle, terminology fixer, missing data completion
