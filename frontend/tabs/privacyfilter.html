Privacy Filter

De-identify medical images, scans, and PDF reports — removes burned-in text, metadata, and document PII, with a built-in editor to manually redact anything the system misses. Runs entirely on CPU.

Note: Processing time is typically 10 to 30 seconds.

Privacy Protocol: This is a non-persistent service. Your data is processed in real-time and purged automatically after use.

Accuracy: Detections may be incomplete or incorrect. Use the Edit button on a preview to manually black out anything missed, then review every output before use.

import requests

    "https://dpi-dev.tanuh.ai/privacy-filter/api/demo-token",

Privacy Filter

                
CPU-based de-identification service that detects and redacts Personally Identifiable Information (PII) and Protected Health Information (PHI) from medical images, DICOM/NIfTI scans, and PDF reports — removing metadata, burned-in pixel text, and document PII while preserving all clinical content.

            


            
Key Capabilities

            

                
FeatureDescription

                
                    
Metadata De-identificationScrubs DICOM tags, EXIF, and NIfTI headers; regenerates DICOM UIDs

                    
Burned-in Text RemovalShape-based connected-component detection + Tesseract OCR find and mask patient text overlaid on the image

                    
PDF / Document RedactionReads the PDF's native text and removes HIPAA Safe Harbor identifiers (names, dates, IDs, contacts) anywhere on the page, plus QR codes & barcodes — true text removal, not a cosmetic overlay

                    
Modality-Aware ScanningFundus, OCT, ultrasound, CT, and MRI layouts handled with format-specific overlay regions, so clinical content is never touched

                    
Manual Review & EditInteractive preview with draw-to-redact boxes, zoom & pan, and one-click removal — catch anything the automated pass misses

                    
Multi-Format SupportDICOM, NIfTI, images (PNG/JPG/TIFF/BMP), and PDF

                    
On-Premises ProcessingAll data stays on the server — no external API calls, fully DPDP Act compliant

                    
Fast ProcessingTypically 10–30 seconds per file on CPU

                
            

        


        
        

            
How to Use

            
Follow these steps to de-identify a file.


            

                

                    
1. Upload File

                    
Navigate to the Privacy Filter tab and click "Choose document to redact...". Upload a DICOM, NIfTI, image, or PDF file. Maximum file size is 25 MB.

                

                

                    
2. Redact File

                    
Click "Redact Document". The system clears metadata, detects burned-in text (CV + OCR), and removes document PII from PDFs. Processing takes 10–30 seconds.

                

                

                    
3. Review & Edit 
                        
                    

                    
Check the detected entities and previews. If anything was missed, click Edit on a preview to open the interactive editor and draw black boxes over it (see Review & Edit).

                

                

                    
4. Download

                    
Download the original and the de-identified output. Redactions are applied to the file itself, in its original format.

                

            


            

                Accuracy Notice
                Detections may be incomplete or incorrect. Please review every output before use in production. The system prioritizes recall (catching more PHI) over precision.
            

        


        
        

            
Review & Edit 
                
                
 
            
Automated de-identification prioritises recall, but no detector is perfect. The interactive editor lets you visually confirm the result and manually redact anything that was missed — for PDFs, images, and DICOM alike.


            
Opening the editor

            
After redaction, each result shows an Original and Redacted preview. Click the Edit button above either preview to open the full-screen editor. Auto-detected redactions appear as boxes you can keep or remove.


            
Editing tools

            

                
ActionHow

                
                    
Add a redactionDrag a rectangle over any PII the system missed — it is black-mapped on apply

                    
Remove a boxClick a box (or its row in the side panel) to delete an unwanted/incorrect redaction

                    
ZoomUse the + / − / Fit controls or Ctrl + scroll to zoom toward the cursor

                    
Pan high-res scansUse the scrollbars, or hold Space and drag, to move around a zoomed image — you can reach every edge

                    
Undo / ClearCtrl+Z to undo, or Clear All to start over

                    
ApplyApply Changes burns the boxes into the file (true removal) and refreshes the preview & download

                
            


            

                Why manual edit matters
                The editor is your safety net for HIPAA Safe Harbor compliance: if any name, ID, date, or other identifier slips through the automated pass, you can remove it in seconds before sharing the file.
            

        


        
        

            
Detection Pipeline

            
The Privacy Filter uses a layered, CPU-only pipeline that removes patient information from both the file's metadata and the image pixels, without altering any clinical content.


            
Layer 1 — Metadata De-identification

            
DICOM tags (patient name, ID, birth date, sex, institution, physician, device, …) are scrubbed and study/series/SOP UIDs regenerated; EXIF and NIfTI header fields are cleared. Pixel and voxel data are left untouched.


            
Layer 2 — Burned-in Text Detection

            
A shape-based connected-component scan locates character-like regions where overlays sit, without reading them. The scan is modality-aware: ophthalmic images (fundus, OCT) use a narrow band inside the black border, while ultrasound / CT / MRI use a full-width header band — so measurements, calipers, and clinical annotations are never masked.


            
Layer 3 — OCR + Rule-Based Classification

            
Tesseract OCR reads the detected regions and deterministic rules classify the text as patient information — names, dates, identifiers, and ages. Confirmed regions are masked with opaque boxes; the redaction is irreversible and leaves surrounding pixels unchanged.


            
Documents (PDF) — Native-Text Safe Harbor Path

            
PDFs take a dedicated, content-based path rather than the pixel pipeline. The PDF's own text layer is read with coordinates and scanned for all HIPAA Safe Harbor identifiers (names, dates & times, phone/email/URL/IP, MRNs and other IDs) anywhere on the page — header banners, body, signatures, footers. Matches are removed with PyMuPDF true redaction (the underlying glyphs are deleted, not just covered), QR codes and barcodes are stripped, and document metadata is scrubbed. Clinical values and reference ranges are preserved.


            
Processing Flow

            

                
1. Load & Format Detection
File type identified and routed to the matching loader (DICOM, NIfTI, image, PDF).

                
2. Metadata Scrub
Header/tag PHI removed; DICOM UIDs regenerated.

                
3. Text-Region Detection
Connected-component scan locates burned-in text in modality-specific regions.

                
4. OCR & Rule Scan
Tesseract reads regions; rules flag names, dates, and IDs.

                
5. Masking
Detected regions blacked out; clinical content preserved.

                
6. Validation & Output
Redacted image re-checked for residual text; de-identified file written in the original format.

Privacy & Data Handling

            
The Privacy Filter is designed with data minimization as a core principle.


            
    
                
  • On-premises only — all processing happens locally, no data leaves the server
    • 
                
  • No storage — uploaded files are deleted immediately after processing
    • 
                
  • No logging — no document content is logged or persisted to disk
    • 
                
  • Anonymous tracking — visitor IPs stored as one-way hashes only
    • 
                
  • DPDP Act compliant — meets India's data protection requirements for data minimization
    • 
            


            

                No External API Calls
                Unlike cloud-based solutions, the Privacy Filter runs entirely on-premises. Your files never leave your infrastructure. All detection runs locally on CPU.
