Key Capabilities

ABDM Schema Validator

Conforms raw texts (diagnoses, procedures, medications) exactly to standard NRCeS/ABDM FHIR StructureDefinition rules.

Multi-Patient Splitter

Uses patient grouping parameters (Age/Sex + date) to auto-split multi-page lab reports into distinct patient bundles.

Structured FHIR Mapping

Translates medical summaries into standardized, ready-to-share FHIR R4 DischargeSummary and DiagnosticReport bundles.

Confidential Offline Execution

Execute on-premises using CPU-optimized binaries for Linux, Mac, or Windows, ensuring zero clinical patient bytes leave the facility.

How It Works

Three key steps running securely on the clinical platform

01

Multi-Engine OCR

Clinical summaries and lab reports are converted using a 6-engine OCR waterfall to handle scans, digital PDFs, and handwriting safely.

02

Patient Grouping

For batch reports, pages are scanned and grouped dynamically based on individual patient parameters to separate patient records cleanly.

03

Graph-Based Extraction

A topological sort workflow extracts clinical details (Diagnoses, MedicationRequests, Observations) and maps them to a validated ABDM Bundle.

100% Confidential On-Premises Analysis

All extraction processes, OCR computation layers, and FHIR mappings compile in memory locally. No data bytes are logged or transmitted externally, maintaining compliance with national healthcare architecture guidelines and privacy protocols.

Download a single executable for your OS. No Python, no pip, no cloud APIs. All data stays on your machine.

Requires: NVIDIA GPU with 16 GB+ VRAM (or Apple Silicon Mac with 16 GB+ memory) and ~20 GB free disk space.

Setup Steps (All Platforms)

1. Install Ollama

Download from ollama.com/download
Linux: curl -fsSL https://ollama.com/install.sh | sh

2. Pull the LLM Model (~16 GB download)

ollama pull gemma4:26b
Requires NVIDIA GPU with 16 GB+ VRAM, or Apple Silicon with 16 GB+ unified memory.

3. Unzip & Make Executable

Unzip the downloaded file. On Linux/Mac run: chmod +x nhcx-extract*
Linux requires sudo apt install libfuse2 if not already installed.

4. Run Clinical Document Extraction

./nhcx-extract abdm report.pdf -o result.json -m gemma4:26b

Use check first to verify setup: ./nhcx-extract check -m gemma4:26b

Privacy: All data stays on your machine. Zero cloud API calls. The output JSON contains a complete FHIR R4 Bundle.

Clinical Document API

Developer API for converting clinical PDFs into ABDM-compliant FHIR DocumentBundles.

Authentication: Uses JWT bearer tokens. Request a 1-day demo token via the form below, then pass it as Authorization: Bearer <token>.
UI Uploads: When using the Clinical Document tab, tokens are fetched automatically.

 Quick Start


        

            
curl

            

                
TOKEN=$(curl -s -X POST https://dpi-dev.tanuh.ai/pdf2abdm/api/token \
  -H "Content-Type: application/json" \
  -d '{"name":"Your Name","email":"you@example.com"}' \
  | python3 -c "import sys,json; print(json.load(sys.stdin)['access_token'])")

curl -X POST https://dpi-dev.tanuh.ai/pdf2abdm \
  -H "Authorization: Bearer $TOKEN" \
  -F "file=@/path/to/clinical_report.pdf"

                
            

        


        

            
Python

            

                
import requests

resp = requests.post(
    "https://dpi-dev.tanuh.ai/pdf2abdm/api/token",
    json={"name": "Your Name", "email": "you@example.com"},
)
token = resp.json()["access_token"]

with open("clinical_report.pdf", "rb") as f:
    r = requests.post(
        "https://dpi-dev.tanuh.ai/pdf2abdm",
        headers={"Authorization": f"Bearer {token}"},
        files={"file": ("report.pdf", f, "application/pdf")},
        timeout=600,
    )
print(r.json())
